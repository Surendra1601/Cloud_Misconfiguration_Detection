How to Use CloudLine

Now that CloudLine is running, here is how to scan your AWS account, view results, and fix security problems.

Running a Security Scan

Simple Analogy Running a scan is like asking a security guard to walk through your entire building and report anything wrong. The guard checks every door, window, and alarm system.

From the Dashboard

Open the Dashboard

Go to http://localhost:5173 (local) or https://app.yourdomain.com (production) in your web browser.

Log In

Enter your username and password. In development mode, any credentials work. In production, use your configured authentication.

View the Dashboard

The main dashboard shows your overall compliance score, severity breakdown, and domain scores. If this is your first visit, the data will be empty until you run a scan.

From the Command Line (API)

You can also run scans using the API directly. This is useful for automation or if you prefer working in the terminal.

Start a new scan

terminal curl -X POST http://localhost:9710/api/v1/scans

This returns a scan ID:

json { "scan_id": "scan_abc123", "status": "running", "started_at": "2026-03-02T10:00:00Z" }

Check scan status

terminal curl http://localhost:9710/api/v1/scans/scan_abc123

View all violations found

terminal curl http://localhost:9710/api/v1/violations

Understanding Your Results

Compliance Score

Your compliance score is a percentage from 0% to 100%. It tells you how many of CloudLine's 21 security checks your account passes.

  • 90-100% — Excellent. Your account is well-secured.
  • 70-89% — Good, but there are some issues to address.
  • 50-69% — Fair. Several security gaps need attention.
  • Below 50% — Needs immediate attention. Many security controls are missing.
terminal # Check your compliance score via API curl http://localhost:9710/api/v1/compliance/score

Violations

A "violation" means CloudLine found something wrong. Each violation includes:

Field What It Means Example
check_id Which security rule was violated CHECK_04
severity How serious the problem is critical
reason Plain English description of the problem "S3 bucket does not block public access"
resource Which AWS resource is affected arn:aws:s3:::my-bucket
domain Which security category data_protection
risk_score Danger level from 0 to 100 92

Severity Levels

Critical

Immediate risk of data breach or account compromise. Fix right away.

Example: S3 bucket is publicly accessible to the entire internet.

High

Significant security weakness. Fix within 24 hours.

Example: RDS database has no encryption.

Medium

Security best practice not followed. Fix within a week.

Example: VPC Flow Logs are not enabled.

Fixing Security Problems

CloudLine gives you three ways to fix problems, from manual to fully automatic.

Tier 1 Follow the Instructions

Every violation comes with three types of fix instructions:

Console Steps

Step-by-step guide to fix it manually in the AWS web console. Best for beginners.

CLI Command

A ready-to-use AWS CLI command you can copy and paste into your terminal.

Terraform Snippet

Infrastructure-as-code that ensures the fix stays in place permanently.

terminal # View remediation for a specific check curl http://localhost:9710/api/v1/\ remediation/suggestions/CHECK_04

Tier 2 One-Click Fix

For 6 common problems, you can fix them with a single API call (or a button click in the dashboard):

Problem Remediation ID What It Fixes
S3 Public Access REM_04 Blocks all public access on the bucket
CloudTrail Logging REM_05 Enables multi-region logging
Open SSH (Port 22) REM_07 Removes 0.0.0.0/0 SSH rule
IMDSv2 Not Enforced REM_08 Requires IMDSv2 on the instance
EBS Not Encrypted REM_17 Enables default EBS encryption
Password Policy REM_02 Sets strong password requirements
terminal # Execute a one-click fix via API curl -X POST \ http://localhost:9710/api/v1/\ remediation/execute \ -H "Content-Type: application/json" \ -d '{ "remediation_id": "REM_04", "resource_arn": "arn:aws:s3:::my-bucket" }'

Tier 3 Auto-Remediation

You can configure CloudLine to automatically fix certain types of violations as soon as they are detected. This is optional and you choose which checks to auto-remediate.

terminal # Enable auto-remediation for S3 public access curl -X PUT \ http://localhost:9710/api/v1/\ remediation/config \ -H "Content-Type: application/json" \ -d '{ "check_id": "CHECK_04", "enabled": true }'
⚠️
Be careful with auto-remediation. Only enable it for checks you fully understand. Auto-remediation changes your AWS resources automatically, which could affect running applications.

Undoing a Fix (Rollback)

If a one-click fix causes problems, you can undo it:

terminal # Rollback a remediation action curl -X POST \ http://localhost:9710/api/v1/\ remediation/rollback \ -H "Content-Type: application/json" \ -d '{ "action_id": "action_xyz789" }'

You can find the action_id in the audit trail:

terminal # View all remediation actions taken curl http://localhost:9710/api/v1/\ remediation/audit

Dashboard Pages

Page URL What You See
Dashboard /dashboard Overall compliance score, severity breakdown, domain scores, risk cards
Violations /violations Sortable table of all findings, filterable by severity/domain/status, click to see details
Trends /trends Line charts showing violations over time (7/30/90 day views)
Executive /executive High-level summary for managers: status banner, KPI cards, domain grid, highest-risk table
Reports /reports Remediation suggestions, audit trail, auto-remediation config

Real-Time Alerts

CloudLine sends alerts through two channels:

Dashboard Notifications

A bell icon in the top-right corner shows live alerts as they happen via WebSocket. Click it to see the feed of recent violations.

Email/SMS via SNS

If configured, CloudLine sends alerts to an SNS topic which can deliver to email, SMS, Slack, or PagerDuty.

API Quick Reference

Method Endpoint What It Does
GET /health Health check
POST /api/v1/scans Start a new scan
GET /api/v1/violations List all violations
GET /api/v1/compliance/score Get compliance score
GET /api/v1/risk/scores Get risk scores
GET /api/v1/risk/summary Risk summary
GET /api/v1/drift/alerts List drift alerts
POST /api/v1/remediation/execute Execute a fix
POST /api/v1/remediation/rollback Undo a fix
GET /api/v1/remediation/audit View fix history

Full API documentation is available at /api/docs (Swagger UI) and /api/redoc (ReDoc) when the backend is running.
