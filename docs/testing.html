Testing Guide

How to verify that all 21 security policies correctly detect misconfigurations. This guide covers running automated tests and understanding what each test proves.

Simple Analogy Testing is like a fire drill. You do not wait for a real fire to see if the alarm works. Instead, you simulate smoke and check that the alarm goes off. CloudLine's tests simulate insecure AWS configurations and verify that the policies detect them.

CloudLine has three types of tests:

1. Policy Tests (OPA/Rego)

66 tests that verify each of the 21 security policies correctly detects misconfigurations. This is what this page focuses on.

opa test policies/ -v

2. Backend Tests (Python)

862 tests that verify the API, pipeline, risk scoring, remediation, and all backend logic works correctly.

make test

3. Frontend Tests (JavaScript)

250 tests that verify the dashboard components, hooks, and API integration work correctly.

make frontend-test

How to Run All Policy Tests

Install OPA

OPA (Open Policy Agent) is the tool that runs the security policies. Install it:

terminal # Linux curl -L -o opa \ https://openpolicyagent.org/\ downloads/latest/opa_linux_amd64_static chmod +x opa sudo mv opa /usr/local/bin/ # macOS brew install opa # Verify installation opa version

Navigate to the project

terminal cd CloudLine

Run all 66 policy tests

terminal opa test policies/ -v

The -v flag shows each test name and its result. You should see 66 tests pass with output like:

output policies/tests/check_01_test.rego: data.aws.check_01_root_account_test.\ test_root_mfa_enabled: PASS (1.234ms) data.aws.check_01_root_account_test.\ test_root_mfa_disabled: PASS (0.567ms) ... --------------------------------------------- PASS: 66/66

Check formatting (optional)

terminal opa fmt --diff policies/

This checks that all policy files follow the standard Rego formatting. If there is no output, everything is formatted correctly.

✅
If all 66 tests pass, every security policy is working correctly and can detect misconfigurations.

How Policy Tests Work

Each test simulates two scenarios:

Secure Configuration (Should Pass)

The test provides a fake AWS resource that is correctly configured. The policy should find zero violations.

"If the door is locked, the alarm should NOT go off."

Insecure Configuration (Should Fail)

The test provides a fake AWS resource that is misconfigured. The policy should find one or more violations with the correct check ID, severity, and reason.

"If the door is unlocked, the alarm SHOULD go off."

Example: S3 Public Access (CHECK_04)

Here is what the test for CHECK_04 looks like, explained line by line:

rego # Test 1: Secure bucket (all blocks on) # Expected: No violations test_all_blocks_enabled if { # Create a fake S3 bucket with ALL # public access blocks turned ON results := check_04.violation with input as { "resource": { "arn": "arn:aws:s3:::my-bucket", "public_access_block": { "block_public_acls": true, "block_public_policy": true, "ignore_public_acls": true, "restrict_public_buckets": true } } } # Should find ZERO violations count(results) == 0 } # Test 2: Insecure bucket (blocks off) # Expected: Violation detected test_public_access_violation if { # Create a fake S3 bucket with public # access blocks turned OFF results := check_04.violation with input as { "resource": { "arn": "arn:aws:s3:::my-bucket", "public_access_block": { "block_public_acls": false, "block_public_policy": false, "ignore_public_acls": false, "restrict_public_buckets": false } } } # Should find at least 1 violation count(results) > 0 # Verify the violation has the right ID some result in results result.check_id == "CHECK_04" result.severity == "critical" }
💡
Key idea: Tests use with input as {...} to provide fake data. No real AWS account is needed. The tests verify the policy logic is correct by checking the results against expected violations.

Automated Testing (CI/CD)

CloudLine runs all tests automatically on every code change via GitHub Actions. The CI pipeline has 5 jobs:

Job What It Tests Must Pass?
OPA Policy Tests Rego formatting + 66 policy tests Yes
Backend Tests Linting + 862 tests + 90% coverage minimum Yes
Frontend Tests Linting + type check + tests + build Yes
Terraform Validate Format check + init + validate Yes
Docker Build Build the backend Docker image Yes

All 5 jobs must pass before code can be merged. This ensures no broken security policies make it to production.

Adding a New Security Check

If you want to add a 22nd check (or modify an existing one), here is the testing workflow:

Write the Rego policy

Create a new file in policies/domain_X/ following the template in the policy guide.

Write the test

Create a test file in policies/tests/. Include at least two tests: one for a secure resource (no violations) and one for an insecure resource (violation detected).

rego # policies/tests/check_22_test.rego package aws.check_22_new_check_test import rego.v1 import data.aws.check_22_new_check # Test: secure resource passes test_secure_resource if { count(check_22_new_check.violation) == 0 with input as { "resource": { "arn": "arn:aws:...", "setting": true } } } # Test: insecure resource is caught test_insecure_resource if { results := check_22_new_check.violation with input as { "resource": { "arn": "arn:aws:...", "setting": false } } count(results) == 1 some r in results r.check_id == "CHECK_22" }

Run the tests

terminal # Run all tests (should be 68 now) opa test policies/ -v # Check formatting opa fmt --diff policies/ # Auto-fix formatting if needed opa fmt -w policies/

Add event mapping and remediation

See the deployment guide for adding event mappings and remediation templates.

Troubleshooting

▶ "command not found: opa"

OPA is not installed. Follow Step 1 above to install it. After installing, run opa version to verify.

▶ Tests fail with "undefined ref"

This usually means a policy file has a typo in the package name. Check that the test's import statement matches the policy's package name exactly.

▶ "opa fmt --diff" shows changes

Your Rego files are not formatted correctly. Run opa fmt -w policies/ to auto-fix the formatting, then re-run the tests to make sure nothing broke.

▶ Backend tests fail

Make sure you have activated the Python virtual environment:

cd CloudLine/backend source venv/bin/activate pip install -e ".[dev]" pytest -v
