Domain 1: Identity & Access Control

5 checks — Who can access your AWS account?
Think of it like These checks are about who has keys to your house, whether the locks are strong enough, and whether anyone has a key they should not.
▶ CHECK_01 Root Account MFA Critical
What This Checks

Whether the root account (the "master key" to your entire AWS account) has Multi-Factor Authentication (MFA) turned on.

Why It Matters

The root account has unlimited access to everything in AWS. If someone steals the password and there is no MFA, they own your entire account. MFA requires a second code (from your phone) in addition to the password.

What Makes It Pass

Root account has MFA enabled (mfa_enabled = true).

What Makes It Fail

Root account does NOT have MFA enabled.

How to Fix
Console (Manual)
  1. Sign in as root user
  2. Go to IAM Dashboard
  3. Click "Add MFA"
  4. Choose Virtual MFA Device
  5. Scan QR code with authenticator app (Google Authenticator, Authy)
  6. Enter two consecutive codes
  7. Click Assign MFA
CLI Command
aws iam create-virtual-mfa-device \ --virtual-mfa-device-name RootMFA \ --outfile /tmp/QRCode.png \ --bootstrap-method QRCodePNG
Compliance Frameworks
CIS AWS 1.5 NIST IA-2(1) PCI DSS 8.3.1
Fix Time: ~10 minutes
▶ CHECK_02 IAM Password Policy High
What This Checks

Whether your AWS account enforces strong password requirements for all IAM users.

Why It Matters

Weak passwords are one of the easiest ways hackers get in. This check ensures passwords are long, complex, and regularly changed.

What Makes It Pass
  • Minimum 14 characters
  • Requires symbols (!@#$%)
  • Requires numbers
  • Requires uppercase and lowercase
  • Remembers last 24 passwords (no reuse)
  • Passwords expire (max age > 0)
What Makes It Fail

Any of the above requirements is not met. Can produce up to 4 separate violations.

How to Fix
CLI Command
aws iam update-account-password-policy \ --minimum-password-length 14 \ --require-symbols \ --require-numbers \ --require-uppercase-characters \ --require-lowercase-characters \ --max-password-age 90 \ --password-reuse-prevention 24

Tier 2 One-click fix available (REM_02)

Compliance Frameworks
CIS AWS 1.8-1.11 NIST IA-5
Fix Time: ~5 minutes
▶ CHECK_03 MFA for All IAM Users High
What This Checks

Whether every IAM user in your account has MFA (two-factor authentication) enabled.

Why It Matters

Even with strong passwords, accounts can be compromised through phishing or password leaks. MFA adds a second layer of protection — like needing both a key and a code to enter.

What Makes It Pass

User has MFA enabled.

What Makes It Fail

User does NOT have MFA enabled.

How to Fix
Console Steps
  1. Go to IAM > Users
  2. Select the user
  3. Click Security Credentials tab
  4. Click Assign MFA Device
  5. Follow the setup wizard
Compliance Frameworks
CIS AWS 1.10 NIST IA-2(1)
Fix Time: ~10 minutes per user
▶ CHECK_10 Unused Credentials High
What This Checks

Whether any IAM users or access keys have not been used in over 90 days.

Why It Matters

Unused credentials are like leaving spare keys under the doormat. If an attacker finds them, they can get in without anyone noticing because the legitimate owner is not watching.

What Makes It Pass

User/key has been used within the last 90 days.

What Makes It Fail
  • User inactive for > 90 days
  • Access key unused for > 90 days
How to Fix
CLI Command
# Deactivate the unused access key aws iam update-access-key \ --user-name jdoe \ --access-key-id AKIAIOSFODNN7EXAMPLE \ --status Inactive
Compliance Frameworks
CIS AWS 1.12 NIST AC-2(3)
Fix Time: ~10 minutes
▶ CHECK_19 IAM Access Analyzer Medium
What This Checks

Whether IAM Access Analyzer is enabled. This AWS service automatically finds resources that are shared with external accounts.

Why It Matters

Without Access Analyzer, you might not realize that an S3 bucket or IAM role is accidentally shared with the public or another account.

What Makes It Pass

At least one Access Analyzer exists.

What Makes It Fail

No Access Analyzers configured.

How to Fix
CLI Command
aws accessanalyzer create-analyzer \ --analyzer-name account-analyzer \ --type ACCOUNT
Compliance Frameworks
CIS AWS 1.20 NIST AC-6
Fix Time: ~5 minutes

Domain 2: Data Protection

6 checks — Is your data encrypted and private?
Think of it like These checks make sure your valuables are in a locked safe (encrypted), not left on the front porch (public), and you have backups in case of fire (disaster recovery).
▶ CHECK_04 S3 Public Access Block Critical
What This Checks

Whether S3 buckets (cloud file storage) have all 4 public access blocking settings enabled.

Why It Matters

A publicly accessible S3 bucket means anyone on the internet can read your files. This has caused massive data breaches at major companies (Capital One, Twitch, etc.).

What Makes It Pass

All 4 public access block settings are enabled:

  • Block public ACLs
  • Block public bucket policies
  • Ignore public ACLs
  • Restrict public buckets
What Makes It Fail

Any of the 4 settings is disabled.

How to Fix
CLI Command
aws s3api put-public-access-block \ --bucket my-bucket \ --public-access-block-configuration \ BlockPublicAcls=true,\ IgnorePublicAcls=true,\ BlockPublicPolicy=true,\ RestrictPublicBuckets=true

Tier 2 One-click fix available (REM_04)

Compliance Frameworks
CIS AWS 2.1.4 NIST AC-3 PCI DSS 1.2.1
Fix Time: ~5 minutes
▶ CHECK_09 RDS Database Security High
What This Checks

Whether RDS databases (managed databases like MySQL, PostgreSQL) are properly secured: not publicly accessible, encrypted, and backed up.

Why It Matters

Databases contain your most sensitive data — user records, passwords, financial data. A public, unencrypted database is a goldmine for attackers.

What Makes It Pass
  • Not publicly accessible
  • Storage is encrypted
  • Backup retention > 0 days
What Makes It Fail

Any of: publicly accessible, not encrypted, or no backups. Can produce up to 3 violations.

How to Fix
CLI Command
aws rds modify-db-instance \ --db-instance-identifier mydb \ --no-publicly-accessible \ --apply-immediately
Compliance Frameworks
CIS AWS 2.3 NIST SC-7 PCI DSS 1.3.1
Fix Time: ~15 minutes
▶ CHECK_11 S3 Encryption High
What This Checks

Whether S3 buckets have server-side encryption enabled.

Why It Matters

Without encryption, files stored in S3 are in plain text. If someone gains unauthorized access, they can read everything. Encryption scrambles the data so it is unreadable without the key.

What Makes It Pass

Encryption is enabled on the bucket.

What Makes It Fail

Encryption is not enabled.

Compliance Frameworks
NIST SC-8 PCI DSS 4.1
Fix Time: ~15 minutes
▶ CHECK_16 Secrets Manager Rotation Medium
What This Checks

Whether secrets (passwords, API keys) stored in AWS Secrets Manager have automatic rotation enabled.

Why It Matters

If a secret is stolen, automatic rotation limits the damage because the secret gets changed regularly. A stolen password that changes every 30 days is far less dangerous than one that never changes.

What Makes It Pass

Rotation is enabled on the secret.

What Makes It Fail

Rotation is not enabled.

Compliance Frameworks
CIS AWS 2.4 NIST IA-5
Fix Time: ~30 minutes
▶ CHECK_17 EBS Volume Encryption High
What This Checks

Whether EBS volumes (virtual hard drives attached to EC2 servers) are encrypted.

Why It Matters

EBS volumes store your server's data. Without encryption, if someone gains access to the underlying storage, they can read all data on the disk.

What Makes It Pass

EBS volume is encrypted.

What Makes It Fail

EBS volume is NOT encrypted.

How to Fix
CLI Command
aws ec2 enable-ebs-encryption-by-default

Tier 2 One-click fix available (REM_17)

Compliance Frameworks
CIS AWS 2.2.1 NIST SC-28 PCI DSS 3.4 HIPAA 164.312
Fix Time: ~5 minutes
▶ CHECK_20 Backup & Disaster Recovery Medium
What This Checks

Whether AWS Backup plans are configured and S3 versioning is enabled for data recovery.

Why It Matters

Without backups, a ransomware attack, accidental deletion, or system failure could mean permanent data loss. Backups are your safety net.

What Makes It Pass

At least one AWS Backup plan exists.

What Makes It Fail
  • No backup plans configured
  • S3 versioning is disabled
Compliance Frameworks
NIST CP-9 HIPAA 164.308
Fix Time: ~20 minutes

Domain 3: Network Security

3 checks — Is your network locked down?
Think of it like These checks are about your fence and gates. Is the front gate wide open? Can anyone walk in from the street? Are the windows locked?
▶ CHECK_07 Security Group — Open SSH/RDP Critical
What This Checks

Whether any security group (firewall rule) allows SSH (port 22) or RDP (port 3389) access from the entire internet (0.0.0.0/0).

Why It Matters

SSH and RDP are remote access protocols. If they are open to 0.0.0.0/0 (everyone on the internet), automated bots will constantly try to break in using brute-force password attacks.

What Makes It Pass

No rules allow SSH or RDP from 0.0.0.0/0.

What Makes It Fail
  • Port 22 (SSH) open to 0.0.0.0/0
  • Port 3389 (RDP) open to 0.0.0.0/0
How to Fix
CLI Command
aws ec2 revoke-security-group-ingress \ --group-id sg-0123456789abcdef0 \ --protocol tcp \ --port 22 \ --cidr 0.0.0.0/0

Tier 2 One-click fix available (REM_07)

Compliance Frameworks
CIS AWS 5.2-5.3 NIST SC-7 PCI DSS 1.3.1
Fix Time: ~5 minutes
▶ CHECK_08 EC2 IMDSv2 Enforcement High
What This Checks

Whether EC2 instances require IMDSv2 (the secure version of the Instance Metadata Service).

Why It Matters

The metadata service gives EC2 instances access to temporary credentials. The old version (IMDSv1) is vulnerable to SSRF attacks — this is exactly how the 2019 Capital One breach happened. IMDSv2 fixes this vulnerability.

What Makes It Pass

http_tokens = "required" (IMDSv2 is enforced).

What Makes It Fail

Instance allows IMDSv1 (the insecure version).

How to Fix
CLI Command
aws ec2 modify-instance-metadata-options \ --instance-id i-0123456789abcdef0 \ --http-tokens required \ --http-endpoint enabled

Tier 2 One-click fix available (REM_08)

Compliance Frameworks
CIS AWS 5.6 NIST AC-4
Fix Time: ~5 minutes
▶ CHECK_15 Network ACL Rules Medium
What This Checks

Whether Network ACLs (NACLs) have overly permissive rules that allow all traffic from the internet.

Why It Matters

NACLs are like the outer fence of your network. If they allow everything in, your security groups (inner doors) are the only defense. Defense in depth means having multiple layers.

What Makes It Pass

No inbound rules allow all protocols from 0.0.0.0/0.

What Makes It Fail

An inbound rule allows all traffic from the internet.

Compliance Frameworks
NIST AC-4 NIST SC-7
Fix Time: ~15 minutes

Domain 4: Logging & Monitoring

4 checks — Are you recording activity?
Think of it like These checks make sure your security cameras are on, recording, and you have someone watching the footage. Without logging, you would never know if someone broke in.
▶ CHECK_05 CloudTrail Configuration Critical
What This Checks

Whether CloudTrail is properly configured to log all API activity across all AWS regions.

Why It Matters

CloudTrail is like the security camera system for your AWS account. It records every action (who did what, when, and where). Without it, you have no way to investigate security incidents.

What Makes It Pass
  • At least one trail exists and is actively logging
  • Trail covers all regions
  • Log file validation is enabled (detects tampering)
What Makes It Fail

No trails, trail not logging, not multi-region, or no log validation. Can produce up to 4 violations.

How to Fix
CLI Command
aws cloudtrail update-trail \ --name my-trail \ --is-multi-region-trail

Tier 2 One-click fix available (REM_05)

Compliance Frameworks
CIS AWS 3.1-3.4 NIST AU-2
Fix Time: ~10 minutes
▶ CHECK_06 VPC Flow Logs High
What This Checks

Whether all VPCs (Virtual Private Clouds) have flow logs enabled to record network traffic.

Why It Matters

Flow logs record who talked to who on your network. Without them, you cannot detect suspicious network activity like data exfiltration or port scanning.

What Makes It Pass

VPC has at least one flow log.

What Makes It Fail

VPC has no flow logs enabled.

Compliance Frameworks
CIS AWS 3.9 NIST AU-12 PCI DSS 10.1
Fix Time: ~15 minutes
▶ CHECK_12 AWS Config Recording Medium
What This Checks

Whether AWS Config is recording all resource configuration changes.

Why It Matters

AWS Config tracks what your resources looked like over time. If a security group was changed, Config tells you what it looked like before and after. This is essential for forensic investigation.

What Makes It Pass
  • Config recorder exists
  • Recorder is actively recording
  • Recording all resource types
What Makes It Fail

No recorder, not recording, or not recording all types.

Compliance Frameworks
CIS AWS 3.5 NIST CM-3
Fix Time: ~15 minutes
▶ CHECK_18 CloudWatch Security Alarms Medium
What This Checks

Whether 10 critical CloudWatch alarms are configured to detect suspicious activity.

Why It Matters

Logging is useless if nobody reads the logs. CloudWatch alarms automatically alert you when suspicious events happen (like unauthorized API calls or root account usage).

Required Alarms (10)
  1. Unauthorized API Calls
  2. Root Account Usage
  3. Console Sign-In Failures
  4. IAM Policy Changes
  5. CloudTrail Changes
  6. Security Group Changes
  7. NACL Changes
  8. Network Gateway Changes
  9. Route Table Changes
  10. VPC Changes
Compliance Frameworks
CIS AWS 4.1 NIST AU-6
Fix Time: ~30 minutes

Domain 5: Threat Detection

2 checks — Are threat detection services active?
▶ CHECK_13 GuardDuty Enabled High
What This Checks

Whether AWS GuardDuty threat detection service is enabled.

Why It Matters

GuardDuty is AWS's built-in threat detector. It uses machine learning to find suspicious activity like cryptocurrency mining, data theft, and compromised credentials. Without it, you are blind to many threats.

What Makes It Pass

At least one GuardDuty detector is enabled.

What Makes It Fail
  • No detectors configured
  • Detector exists but is disabled
How to Fix
CLI Command
aws guardduty create-detector \ --enable \ --finding-publishing-frequency \ FIFTEEN_MINUTES
Compliance Frameworks
CIS AWS 4.15 NIST SI-4
Fix Time: ~5 minutes
▶ CHECK_14 Lambda Function Security Medium
What This Checks

Whether Lambda functions (serverless code) have X-Ray tracing enabled and encrypt environment variables.

Why It Matters

Lambda functions often contain database passwords and API keys in environment variables. Without KMS encryption, these secrets can be exposed. X-Ray tracing helps detect performance issues and errors.

What Makes It Pass
  • X-Ray tracing is Active
  • Environment variables are encrypted with KMS
What Makes It Fail
  • Tracing not enabled
  • Environment not encrypted
Compliance Frameworks
NIST AC-6 NIST SC-28
Fix Time: ~20 minutes

Cross-Resource: Capital One Pattern

1 check — Detects the combination that caused the Capital One breach
⚠️
This is the most advanced check. Instead of looking at one resource at a time, it looks at how multiple resources combine to create a dangerous pattern.
▶ CROSS_01 Capital One Breach Pattern Critical
What This Checks

Whether any EC2 instance has the same dangerous combination that enabled the 2019 Capital One data breach: a public IP + an overpermissive IAM role + the old metadata service (IMDSv1).

Why It Matters

In 2019, a hacker exploited a web application firewall vulnerability to access the EC2 metadata service, stole IAM role credentials, and used them to download 100+ million customer records from S3. This check detects that exact pattern.

What Makes It Pass

No EC2 instance has all three risk factors simultaneously.

What Makes It Fail

An EC2 instance has ALL of:

  • A public IP address
  • An IAM role with overpermissive policies (AdministratorAccess, AmazonS3FullAccess, PowerUserAccess, or IAMFullAccess)
  • IMDSv1 enabled (http_tokens is not "required")
How to Fix

Break the chain by fixing any one of the three factors:

  1. Best: Enforce IMDSv2 on the instance
  2. Replace overpermissive IAM roles with least-privilege policies
  3. Remove the public IP if the instance does not need internet access
Compliance Frameworks
CIS AWS 1.16 CIS AWS 5.2 NIST AC-6
Fix Time: ~60 minutes
